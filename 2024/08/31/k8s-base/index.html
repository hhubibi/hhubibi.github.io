k8s学习笔记

容器原理

容器的本质是单进程模型

为什么容器比hypervisor更轻量？

  • 虚拟机要运行guest os，有额外的资源开销，虚拟机自身就要占用内存
  • 虚拟机里的程序对host os的调用需要经过vmm层，又是一层性能开销
  • 但是容器的隔离性不如虚拟机强

cgroups和namespace

  • namespace：用于资源隔离，包括pid，mount，uts，ipc，network和user这几类，可以通过clone，unshare和setns设置namespace
  • cgroups：用于资源限制，主要是硬件（cpu，memory，网络带宽，磁盘等）

为什么容器里执行top看到的是宿主机的CPU和内存信息？

因为top看的是/proc目录下记录内核运行状态的一系列特殊文件，用户可以通过这些文件看到正在运行的进程信息，包括cpu吕永利，内存占用率等。但是/proc文件系统不知道cgoups对容器做了什么限制。

容器创建

  • 启用linux namespace配置
  • 设置cgroups参数
  • 切换进程的根目录（chroot）

rootfs

包含了一个操作系统的文件、配置和目录，但是不包括内核。rootfs使容器具有一致性，在云端和本地都能运行。

增量制作rootfs，引入layer的概念，使用UnionFS，将不同位置的目录联合挂载到一个目录。

rootfs自底向上由三部分组成

  • 只读层：ro(read only) + wh(whiteout)
    • whiteout：支持只读层的删除，在可读写层创建一个.wh.${filename}的whiteout文件，做到遮挡的作用
  • init层：ro + wh，用来存放/etc/hosts和/etc/resolv.conf，用户会在启动容器的时候写入一些指定的值，但是不希望被docker commit，因此单独成一个层，commmit的时候只会提交可读写层
  • 可读写层：rw，增删改操作

volumn

支持宿主机的文件和目录映射到容器中。原理是Linux的bind mount（绑定挂载技术），其实就是把容器目录的dentry重定向到宿主机目录，但是宿主机其实是无感的（因为mount namespace的隔离），docker commit不会提交这个目录的东西。

k8s

  • Master（控制节点）
    • api server：和用户交互，提供认证、授权，Api注册和发现
    • scheduler：调度，将pod调度到node上
    • controller manager：负责容器编排
  • 计算节点
    • kubelet：负责维护pod生命周期，创建、更新和销毁
    • kubeproxy：集群内部服务发现和负载均衡
    • 容器运行时：docker，containerd，cri-o
    • device plugin

k8s特点

  • 任务编排
  • 声明式api
  • 统一的方式抽象底层Infra（计算、存储、网络等）
  • 调度的最小单位为pod，pod可以有1或多个容器，共享namespace、网络和存储资源，其实就是一个任务组
  • 弹性伸缩
  • 滚动更新
  • 服务发现和负载均衡

pod

一组共享了某些资源的容器，网络、存储，ns

pod中infra容器（资源最少，轻量）最先启动，用来hold住资源，使pod中的容器保持对等关系，不需要一个先于另一个启动。pod的生命周期和infra一致。

其实是一种sidecar，在一个Pod中启动一个辅助容器来完成一些独立于主进程容器的工作。

pod状态

  • pending：api对象已创建并被保存到etcd中，但是有容器没有被顺利创建
  • running：pod已经调度成功和一个节点绑定，容器都创建成功了，至少有一个在运行
  • succeeded：容器正常运行完毕
  • failed：至少有一个容器以不正常的状态退出
  • unknown：可能主从节点间的通信出问题了
